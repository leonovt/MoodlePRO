MoodlePRO — Summary & Quiz Feature Plan

Per-item summaries, course-wide scoped summaries, and video topic chaptering

📋 Status: planning complete, build in progress. This page describes what was approved and is about to be built — not yet a results report. A separate FEATURE_STATUS.html will be generated once implementation and tests finish.

What's being built

1

Per-item Summary + Quiz

A button appears next to every course activity (assignment, resource, forum, zoom, etc.) on the course page. Click it to get a summary and a quiz for that single item.

2

Course-wide scoped summary

On the courses dashboard, each course gets a "Summarize" button with a scope picker: Everything, Assignments only, or Lectures only.

3

Video topic chapters

Once a lecture's transcript is ready, it's split into topic "chapters." Each chapter gets its own Summary and Quiz button, and clicking a chapter seeks the video to that point.

Server changes

FileChange
app/services/summarizer.py new FakeSummarizer — deterministic canned summary, no real LLM call
app/services/quiz.py new file QuizGenerator ABC + FakeQuizGenerator — 3 canned multiple-choice questions
app/services/chaptering.py new file Chapterer ABC + FakeChapterer — groups a job's segments into topic chapters with real timestamps

New endpoints

MethodPathPurpose
POST/items/summaryFake summary for one scraped item
POST/items/quizFake quiz for one scraped item
POST/courses/summaryAggregated fake summary+quiz over a scope-filtered item list
GET/jobs/{job_id}/chaptersFake topic chapters for a finished transcription job
POST/jobs/{job_id}/chapters/{chapter_id}/summaryFake summary scoped to one chapter
POST/jobs/{job_id}/chapters/{chapter_id}/quizFake quiz scoped to one chapter

The server never logs into Moodle itself — it has no session cookie. The extension scrapes visible page text and POSTs it; the server just runs the (currently fake) summarizer/quiz/chaptering logic over whatever text it's given.

Build & verify steps

  1. Agent A (server) implements the 3 fake services, schemas, router, wiring, and test_content.py, then runs pytest itself.
  2. Agent B (extension) implements the modal, scraper, course-items, course-toolbar, chapters modules, the inject.js router, and 4 new vitest files, then runs npx vitest run itself.
  3. Both run in parallel — they touch disjoint directories, so there's no merge conflict risk.
  4. I review both diffs and re-run the full test suite (server + gpu_worker + extension) to confirm nothing regressed.
  5. I generate FEATURE_STATUS.html with pass/fail counts and sample fake JSON output from each new endpoint.
